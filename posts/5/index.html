We have to compromise “persistence agnostic domain model” principle.Â  This is happened even with popular frameworks ActiveRecord (Rails) and Entity Framework (.NET) too.Â  I skip the compromise part as of now.

In Azure, you have two choices to persist objects.Â  One is table storage and another one is SQL Azure.Â  Typically, Web 2.0 applications use mixed approach like frequently used read-only data on NOSQL and source of truth is on relational data stores (CQRS).Â  This would be recommended when your application is running on Cloud, because every byte is billable and metered.Â  In this article, I brief the concerns when you are choosing Table Storage as source of truth. Â  Would Azure Table Storage be a good choice for domain object repository?Â  Though this is not the time to say “Azure Table Storage for domain object-repository should be highly recommended, partially or completely avoided” kind of opinions, but I can share some of the experiences with Azure Table Storage.

Due to high performance, Azure axes some NOSQL features those are typically available in other NOSQL products.Â  This seems like to get the ambitious mileage advertised for a bike, bike companies said you should drive on such a road, wheather and load conditions.

Is Key-Value Data Store Enough?

Key-Value based data stores are the actual starting point of NOSQL revolution, later Document based data store has been widely adopted for object persistence.Â  Document data store has the capability to persist an object (complex data type) against a key, however Key-Value data store can persist only scalar values.Â  This means that the Key-Value’s entity model is very much like relational representation (primary key – foreign key and link table), instead in Document data store, we have the choice to embed object into another object.Â  In the above example, Customer’s Order object can be embedded within Customer, and OrderLine and Product would be by reference. However, Azure Table Storage is just Key-Value data store.Â  You still have to provide meta-data for referential integrity.

Am I smart enough on “Partition Key” decision?

Physical location of a table in Azure (and all other NOSQL data stores too) is based on Partition Key (In Mongo DB, Sharding Key) selection.Â  You can get the “indexing” like faster query result only when you give partition key and row key.Â  Hence, the partition key selection is one of the architectural decision for a cloud application.Â  How smart you are to choose the right partition key is matter here!Â  In the Customer-Order, we can simply choose the following as partition key for the respective tables:

  • Customer – first letter of Name in upper case
  • Product – first letter of Name in upper case
  • Order – again use the partition key of the customer, since Order is always made by a customer.
  • OrderLine – either use Order table partition key or Product table partition key.Â  If we choose Order table, the OrderLine table partition will use Customer table partition key.

The below figure depicts how these tables would be distributed in a data center with four nodes.

Now the concerns are:

Is table partitioning happening always or based on capacity? Typically, NOSQL data stores (like Mongo) starts “sharding” when running out of drive capacity in the current machine. It seems very optimal way.Â  However, there is no clear picture on Azure table storage’s sharding.

Data store is even smarter than me when sharding.Â  In Mongo DB, it only asks table object property or properties as as “Sharding” key.Â  Based on the load, these data stores scales out data across servers.Â  The sharding algorithm intelligently split the entities between the available servers based on the values in the provided “Sharding” key.Â  However, Azure asks the exacts value in the partition key and it groups entities those have same value.Â  Azure does not give the internals of how does partitioning happens.Â  Will it scale-out on all the nodes in the data center? or limit to some numbers?Â  No clear pictures though.

What will happen if entities with same partition key on the single table server run out storage capacity? Do not have a clear picutre.Â  Some papers mentioned that table server is the abstraction and called as “access layer”, which in turns has “persistent storage layer” which contain scalable storage drives. The capacity of the drives will be increased based on the current capacity of the storage for a table server.

Interestingly, I found a reply in one of the Azure forum for a similar question (but quoted couple of years before):

…our access layer to the partitions is separated from our persistent storage layer.Â  Even though we serve up accesses to a single partition from a single partition server, the data being stored for the partition is not constrained to just the disk space on that server.Â Â  Instead, the data being stored for that partition is done through a level of indirection that allows it to span multiple drives and multiple storage nodes.

Since, based on the size of data we don’t know how many actuals table servers will be created for a table.Â  Assume, a business which needs more Orders for a limited Products and small Customers should have largely scaled out Order and OrderLine table.Â  But, here these two tables are restricted by small Customer table.Â  A web 2.0 company may need more products and customers which linearly have very large Order and OrderLines.Â  If this scaled out on large number of table servers, co-location between OrderLine and Product should be important as well for Customer-Order association.Â  IfÂ  Azure table storage scale-out algorithm does the partitioning based on the knowledge of that table only, then unnecessary network latency will be introduced.

Is ADO.NET Data Service Serializer enough for the business?

Enumeration are very common in domain models.Â  However, ADO.NET data service serializer does not serialize it.Â  Either we need to remove enumerated properties or to write custom serializer.
